# DevFlow Engine 阶段输入输出与 Agent 设计

## 设计原则

当前版本使用 Skill-driven LLM Agent。Pipeline 引擎负责阶段流转、上下文传递和人工检查点；每个 Agent 基于对应 Skill 输出真实阶段产物，用来验证“输入需求 - 需求澄清 - 阶段流转 - 人工审批 - 回退重跑 - 最终交付”的闭环。

## 阶段表

| 阶段 | 输入 | 负责 Agent | 处理内容 | 输出 | 人工审批 |
|---|---|---|---|---|---|
| 需求分析 | 用户输入的自然语言需求；历史 Reject 原因 | 需求分析 Agent | 按 SCQA 澄清背景、痛点、边界和验收标准；输出待澄清问题、低保真原型说明和审核建议 | 结构化需求文档 + 原型预览 + Checkpoint 建议 | 是 |
| 方案设计 | 需求分析产物；如有 Reject，则加入驳回原因 | 方案设计 Agent | 生成技术方案、涉及模块、数据结构变化、UI 变化和风险点 | 技术方案文档 | 是 |
| 代码生成 | 已审批的技术方案 | 代码生成 Agent | 生成代码变更说明和 diff 摘要 | 修改文件列表、diff 摘要、关键实现说明 | 是 |
| 测试生成 | 需求分析产物 + 代码生成产物 | 测试生成 Agent | 生成测试用例、测试范围和模拟测试结果 | 测试报告 | 是 |
| 代码评审 | 技术方案 + 代码变更 + 测试结果 | 代码评审 Agent | 检查正确性、稳定性、代码质量和交付风险 | 评审报告 | 是 |
| 交付总结 | 全部阶段产物 | 交付总结 Agent | 汇总本次需求交付内容 | 变更摘要、测试摘要、MR 描述草稿 | 是 |

## 人工审批规则

当前包含六个 Human-in-the-loop 卡点，每个阶段产出后都必须等待人工确认：

1. 需求分析后审批  
   Approve 后进入方案设计；Reject 时必须填写原因，原因可包含对澄清问题的回答，并回到需求分析阶段重新生成。

2. 方案设计后审批  
   Approve 后进入代码生成；Reject 时必须填写原因，并回到方案设计阶段重新生成。

3. 代码评审后审批  
   Approve 后进入交付总结；Reject 时保留原因，后续可扩展为回到代码生成阶段重新生成。

4. 代码生成、测试生成、交付总结后审批  
   Approve 后进入下一阶段；Reject 时必须填写原因，并回到当前阶段重新生成。交付总结 Approve 后，Pipeline 标记为 completed。

## Demo 需求

默认 Demo 需求为：

> 给任务管理系统增加按优先级筛选任务的功能。

这个需求会被 6 个 Mock Agent 依次处理，用来演示 AI 研发流程引擎的最小闭环。
