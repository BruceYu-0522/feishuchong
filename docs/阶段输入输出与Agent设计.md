# DevFlow Engine 阶段输入输出与 Agent 设计

## 设计原则

第一版先使用 Mock Agent。每个 Agent 不直接调用真实大模型，而是用模板产出稳定结果，用来验证 Pipeline 是否能完成“输入需求 - 阶段流转 - 人工审批 - 回退重跑 - 最终交付”的闭环。

## 阶段表

| 阶段 | 输入 | 负责 Agent | 处理内容 | 输出 | 人工审批 |
|---|---|---|---|---|---|
| 需求分析 | 用户输入的自然语言需求 | 需求分析 Agent | 拆分用户故事、功能范围、验收标准和边界情况 | 结构化需求文档 | 否 |
| 方案设计 | 需求分析产物；如有 Reject，则加入驳回原因 | 方案设计 Agent | 生成技术方案、涉及模块、数据结构变化、UI 变化和风险点 | 技术方案文档 | 是 |
| 代码生成 | 已审批的技术方案 | 代码生成 Agent | 生成代码变更说明和 diff 摘要 | 修改文件列表、diff 摘要、关键实现说明 | 否 |
| 测试生成 | 需求分析产物 + 代码生成产物 | 测试生成 Agent | 生成测试用例、测试范围和模拟测试结果 | 测试报告 | 否 |
| 代码评审 | 技术方案 + 代码变更 + 测试结果 | 代码评审 Agent | 检查正确性、稳定性、代码质量和交付风险 | 评审报告 | 是 |
| 交付总结 | 全部阶段产物 | 交付总结 Agent | 汇总本次需求交付内容 | 变更摘要、测试摘要、MR 描述草稿 | 否 |

## 人工审批规则

第一版包含两个 Human-in-the-loop 卡点：

1. 方案设计后审批  
   Approve 后进入代码生成；Reject 时必须填写原因，并回到方案设计阶段重新生成。

2. 代码评审后审批  
   Approve 后进入交付总结；Reject 时保留原因，后续可扩展为回到代码生成阶段重新生成。

## Demo 需求

默认 Demo 需求为：

> 给任务管理系统增加按优先级筛选任务的功能。

这个需求会被 6 个 Mock Agent 依次处理，用来演示 AI 研发流程引擎的最小闭环。
