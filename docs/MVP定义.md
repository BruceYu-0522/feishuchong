# DevFlow Engine MVP 定义

## 一句话定义

DevFlow Engine 是一个 AI 驱动的研发流程引擎。用户输入一个功能需求后，系统会组织多个 Agent 按研发流程依次完成需求分析、方案设计、代码生成、测试生成、代码评审和交付总结，并在关键节点提供人工审批与回退能力。

## 项目想解决的问题

传统 AI 编程工具通常只帮助开发者完成某一个局部任务，例如生成一段代码、解释一个函数或补全测试用例。但真实的软件研发不是单点动作，而是一条从需求理解到最终交付的完整链路。

本项目希望证明：AI 不只是代码助手，也可以被编排进完整研发流程中，像一个小型研发团队一样，按照固定阶段协作完成一次需求交付。

## Demo 需求

第一版演示选择一个足够小、容易验证的需求：

> 给任务管理系统增加“按优先级筛选任务”的功能。

选择这个需求的原因：

- 功能边界清晰，容易讲明白。
- 可以覆盖需求分析、方案设计、代码变更、测试和评审。
- 不需要复杂业务背景，评委能快速理解。
- 适合作为演示用的目标代码库功能变更。

## MVP 流程

第一版 Pipeline 包含以下阶段：

1. 需求分析
2. 方案设计
3. 人工审批
4. 代码生成
5. 测试生成
6. 代码评审
7. 交付总结

## 阶段产物

### 1. 需求分析

输入：用户的自然语言需求。

输出：

- 用户故事
- 功能范围
- 验收标准
- 边界情况

### 2. 方案设计

输入：需求分析结果。

输出：

- 技术方案
- 涉及文件或模块
- 数据结构变化
- UI 变化说明
- 风险点

### 3. 人工审批

输入：方案设计产物。

输出：

- Approve：进入代码生成阶段
- Reject：填写驳回原因，并回到方案设计阶段重新生成

### 4. 代码生成

输入：已通过审批的技术方案。

输出：

- 代码 diff 摘要
- 修改文件列表
- 关键实现说明

第一版可以先使用模板或模拟 Agent 生成代码变更说明，不要求一开始就真实修改复杂代码库。

### 5. 测试生成

输入：需求分析结果和代码生成结果。

输出：

- 测试用例
- 测试范围
- 模拟测试结果
- 未覆盖风险

### 6. 代码评审

输入：技术方案、代码生成结果和测试结果。

输出：

- 正确性检查
- 安全性或稳定性风险
- 代码质量建议
- 是否建议进入交付

### 7. 交付总结

输入：全部阶段产物。

输出：

- 本次需求完成内容
- 变更摘要
- 测试摘要
- MR / PR 描述草稿
- 后续优化建议

## 人工检查点

第一版至少包含两个人工检查点：

1. 方案设计后审批  
   用户可以确认方案是否合理。如果不合理，可以 Reject 并填写原因，例如“方案没有考虑空状态”。

2. 代码评审后审批  
   用户可以确认本次变更是否可以交付。如果不通过，可以 Reject 并要求回到代码生成或方案设计阶段。

## 第一版目标

MVP 不追求一次性实现完整的真实 AI 编程平台，而是优先实现一个可以运行、可以演示、可以解释清楚的闭环。

第一版必须做到：

- 可以输入一个需求。
- 可以启动 Pipeline。
- 可以看到每个阶段的状态变化。
- 可以查看每个阶段的产物。
- 可以在检查点 Approve 或 Reject。
- Reject 后可以携带原因回退并重新生成。
- 最终可以生成交付总结。

第一版可以暂时不做：

- 真实创建 Git 分支。
- 真实提交 MR / PR。
- 真实修改大型代码库。
- 多模型 Provider 切换。
- 复杂多 Agent 并行协作。

## 推荐实现方式

### 前端

做一个 DevFlow 控制台，包含：

- 需求输入框
- Pipeline 阶段列表
- 当前阶段状态
- 阶段产物展示区
- Approve / Reject 操作区
- 最终交付总结区

### 后端

提供 API 管理流程状态：

- 创建 Pipeline
- 启动 Pipeline
- 查询 Pipeline 状态
- 提交审批结果
- 查询阶段产物

### Agent

第一版先使用 Mock Agent，也就是用模板函数模拟不同 Agent 的输出：

- 需求分析 Agent
- 方案设计 Agent
- 代码生成 Agent
- 测试生成 Agent
- 代码评审 Agent
- 交付总结 Agent

等流程跑通后，再把部分 Mock Agent 替换成真实大模型调用。

## 演示主线

演示时按照下面的故事讲：

1. 我输入一个需求：给任务管理系统增加按优先级筛选任务。
2. 系统自动进入需求分析阶段，生成用户故事和验收标准。
3. 系统进入方案设计阶段，生成技术方案。
4. 我在方案审批点故意 Reject 一次，填写“缺少空状态处理”。
5. 系统带着驳回原因重新生成方案。
6. 我 Approve 方案，系统继续生成代码变更说明。
7. 系统生成测试用例和模拟测试结果。
8. 系统完成代码评审。
9. 我通过最终审批，系统生成交付总结和 MR 描述草稿。

## 成功标准

如果评委能从演示中看明白下面三点，第一版就是成功的：

1. AI 被拆分成多个明确角色，而不是一个泛泛聊天机器人。
2. 需求到交付的流程被 Pipeline 管理起来了。
3. 人可以在关键节点介入、审批、驳回并驱动流程重跑。
