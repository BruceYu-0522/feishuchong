# DevFlow Engine 需求要点、阶段拆解与 Must-have 清单

## 1. 需求要点

本项目要求设计并实现一个 AI 驱动的研发流程引擎 DevFlow Engine。它的重点不是让 AI 单独生成一段代码，而是把 AI 编排进完整的软件研发流程中，让平台能够从一个自然语言需求出发，依次完成需求分析、方案设计、代码生成、测试生成、代码评审和交付总结。

项目核心目标可以概括为：

> 证明 AI 可以被组织成多个明确分工的 Agent，并在 Pipeline 管理下完成一次从需求到交付的研发闭环。

第一版 Demo 需求选定为：

> 给任务管理系统增加“按优先级筛选任务”的功能。

选择该需求的原因是功能边界清晰、演示成本较低、容易覆盖完整研发链路，也方便评委理解系统运行过程。

## 2. 阶段拆解

第一版 Pipeline 拆解为 7 个阶段。

### 阶段一：需求分析

目标：把用户输入的自然语言需求转成结构化需求。

输入：

- 用户输入的功能需求

输出：

- 用户故事
- 功能范围
- 验收标准
- 边界情况

### 阶段二：方案设计

目标：根据结构化需求生成技术实现方案。

输入：

- 需求分析结果

输出：

- 技术方案
- 涉及模块或文件
- 数据结构变化
- UI 变化说明
- 风险点

### 阶段三：方案人工审批

目标：让人类在关键节点判断方案是否可以继续执行。

输入：

- 方案设计产物

输出：

- Approve：进入代码生成阶段
- Reject：填写驳回原因，回到方案设计阶段重新生成

### 阶段四：代码生成

目标：根据已审批的方案生成代码变更结果。

输入：

- 已通过审批的技术方案

输出：

- 修改文件列表
- 代码 diff 摘要
- 关键实现说明

第一版可以先使用 Mock Agent 或模板产出代码变更摘要，不要求一开始就真实修改大型代码库。

### 阶段五：测试生成

目标：根据需求和代码变更生成测试方案。

输入：

- 需求分析结果
- 代码生成结果

输出：

- 测试用例
- 测试范围
- 模拟测试结果
- 未覆盖风险

### 阶段六：代码评审

目标：检查代码变更是否具备交付质量。

输入：

- 技术方案
- 代码变更结果
- 测试结果

输出：

- 正确性检查
- 稳定性风险
- 代码质量建议
- 是否建议交付

### 阶段七：交付总结

目标：整合全部阶段产物，生成最终交付说明。

输入：

- 全部阶段产物

输出：

- 需求完成说明
- 变更摘要
- 测试摘要
- MR / PR 描述草稿
- 后续优化建议

## 3. Must-have 清单

### 3.1 Pipeline 引擎

- 支持定义多个研发阶段。
- 支持阶段按顺序流转。
- 支持记录每个阶段的状态。
- 支持把上一阶段输出作为下一阶段输入。
- 支持启动、暂停、继续和结束 Pipeline。

### 3.2 Agent 编排

- 每个阶段绑定一个对应的 Agent。
- 每个 Agent 有明确职责。
- 每个 Agent 有清晰的输入和输出格式。
- 第一版可以使用 Mock Agent 模拟输出。
- 后续可以将部分 Mock Agent 替换为真实大模型调用。

### 3.3 Human-in-the-loop 人工检查点

- 至少包含 2 个人工检查点。
- 支持 Approve 继续流程。
- 支持 Reject 回退流程。
- Reject 时必须填写原因。
- 回退后，系统需要携带驳回原因重新生成产物。

### 3.4 API-first 架构

- 核心操作需要通过 API 暴露。
- 至少包含创建 Pipeline、启动 Pipeline、查询状态、提交审批、查看产物等接口。
- 后续可以补充 Swagger / OpenAPI 文档。

### 3.5 可运行的端到端 Demo

- 用户可以输入一个需求。
- 系统可以完整跑完一次 Pipeline。
- 页面可以展示每个阶段的状态和产物。
- 用户可以在检查点进行审批或驳回。
- 最终可以生成交付总结。

## 4. 第一版暂不做内容

为了优先完成可演示闭环，第一版暂不强制实现以下内容：

- 真实创建 Git 分支。
- 真实提交 MR / PR。
- 真实修改大型代码库。
- 多模型 Provider 切换。
- 多 Agent 并行协作。
- 复杂代码库语义索引。
- 浏览器插件或网页圈选改代码能力。

## 5. 当前阶段结论

本阶段已经完成了对赛题的初步理解和 MVP 范围收敛。项目第一版应优先围绕“任务管理系统优先级筛选”这个小需求，完成从需求输入到交付总结的 AI 研发流程闭环。

后续开发重点不是一开始追求 Agent 能力复杂，而是先让 Pipeline、阶段产物、人工审批和回退机制跑通。只要这个闭环能清晰演示，就能体现项目对 AI Native 研发流程的理解。
